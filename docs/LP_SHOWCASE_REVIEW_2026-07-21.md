# PAOPAO Showcase LP レビュー記録

## 前提

目的は、ホームページ制作を説明するだけでなく、ページそのものを制作能力の証拠にすること。
レビュー担当は実在の人物ではなく、判断基準を分離した5つの仮想視点。各視点で別々に問題を抽出し、共通項だけでなく相反する要求も記録する。

## 第1段階: 構想・初期実装レビュー

### 1. 依頼を検討する小規模事業者
- 最初の数秒でホームページ制作が主役だと分かる必要がある。
- 自動化と業務整理は補助サービスに下げる。
- 無料制作枠の条件を曖昧にしない。

### 2. クリエイティブディレクター
- 黒背景と角丸カードだけでは既視感が強い。
- 立体オブジェクトは装飾ではなく、制作力を実演する中心表現として置く。
- 均等カードを避け、編集的な縦方向のリズムを作る。

### 3. UX・コンバージョン担当
- 「すごい」で終わらず、サービス理解と相談へ自然につながる必要がある。
- WEB / AUTOMATION / OPERATIONSを触って切り替えられる実演にする。
- 相談時に送る内容を3項目に絞り、コピー機能を付ける。

### 4. フロントエンド実装担当
- 外部ライブラリへ依存せず、WebGL非対応時のフォールバックを用意する。
- スマホでは解像度とFPSを抑え、非表示タブでは描画を止める。
- reduced-motion、キーボード操作、コンテキストロストへ対応する。

### 5. 厳しい発注者
- 「できます」という文言より、動く成果物を先に見せる。
- 経歴は年数や肩書きではなく、制作へどう生きるかを説明する。
- このLPを最初の制作事例として成立させる。

## 第1段階の修正

- ファーストビューを「ホームページそのものを、最初の実績に。」へ変更。
- 外部ライブラリなしのWebGLレイマーチングによる精密オブジェクトを実装。
- ポインターとスクロールに反応する立体表現を追加。
- WEB / AUTOMATION / OPERATIONSの3デモを実装。
- ホームページ制作を主役、自動化と業務整理を補助へ再配置。
- 経歴を制作能力への変換として再編集。
- 相談テンプレートのコピー機能を追加。
- PC、タブレット、スマホで、存在確認ではなく実表示状態を検査するQAへ更新。

## 第2段階: 実画面レビュー

GitHub ActionsのPC・タブレット・スマホ画像を5視点で確認した。

### 1. 小規模事業者
- 主力サービスは明確になった。
- 経歴が「だから何ができるか」まで説明されている。
- 無料制作枠から相談までの導線も理解できる。

### 2. クリエイティブディレクター
- WebGLオブジェクトは既存LPとの差別化として成立。
- ヒーロー見出しの最終行が不自然に折り返され、完成度を下げていた。
- 下層セクションの黒い空白は演出ではなく欠陥に見える。

### 3. UX・コンバージョン担当
- デモ切り替えは意味があり、単なる装飾ではない。
- スクロールアニメーションの対象が未表示のまま残ると情報が存在しないように見える。
- JavaScriptやIntersectionObserverが遅れても内容は見えるべき。

### 4. フロントエンド実装担当
- WebGLは正常に描画されたが、ソフトウェアレンダリング環境では負荷が高い。
- タブレット相当でフレーム間隔と長時間タスクが増加。
- デモ切り替え計測にページのスムーズスクロール時間が混入していた。

### 5. 厳しい発注者
- ファーストビューは「こんなこともできるのか」という入口になった。
- ただし黒い空白が一度でも見えれば、完成品としての信用は失われる。
- 見た目だけでなく低性能環境でも破綻しないことが必要。

## 第2段階の修正

- reveal要素を常時表示へ変更し、動きは位置変化だけに限定。
- ヒーロー最終行の文字サイズと改行を調整。
- スマホの不要なスクロール案内を削除。
- WebGLのレイマーチ回数、描画解像度、FPSを端末幅別に削減。
- オブジェクトが画面外へ出たら描画を停止。
- デモ切り替えの計測からスムーズスクロール時間を除外。

## 第3段階: 最適化後の再レビュー

Visual QAとPerformance QAはPC・タブレット・スマホの全環境で成功した。実画面を再度5視点で確認した。

### 1. 小規模事業者
- ホームページ制作が主役であることは迷わない。
- 実演、サービス、経歴、進め方、相談の順番も理解しやすい。
- スマホで実演の一部が切れると、提供品質への不安につながる。

### 2. クリエイティブディレクター
- 立体オブジェクトと編集的なレイアウトはショーケースとして成立。
- スマホの「最初の実績に。」が最後の一文字だけ次行へ落ちていた。
- 相談見出しも過剰に細かく折り返され、終盤の力を弱めていた。

### 3. UX・コンバージョン担当
- デモタブが横スクロールになり、WEBタブが画面外へ隠れる状態があった。
- Operationsデモの下部が固定高の画面内で切れていた。
- 3つの選択肢はスマホでも最初から全部見える方がよい。

### 4. フロントエンド実装担当
- ready timeはPC 667ms、タブレット1037ms、スマホ325ms。
- エラー、404、横スクロール、外部スクリプトは0件。
- WebGLは画面外停止と低解像度化によりQA基準内へ収まった。
- 固定高とabsolute配置はスマホの可変コンテンツに不向き。

### 5. 厳しい発注者
- 相談テンプレートボタンの文字が暗く、無効状態に見えた。
- 技術的な驚きがあっても、基本的な可読性の欠陥が残れば公開不可。
- 「高級感」は暗色や立体物ではなく、細部が破綻しないことで成立する。

## 第3段階の修正

- スマホのヒーロー最終行を縮小し、一行で意図的に収める。
- 実演タブをスマホで3列表示し、全項目を最初から見せる。
- スマホでは選択中パネルだけを通常フローで表示し、高さを内容に合わせる。
- Operationsデモの切れを解消。
- 相談テンプレートボタンの文字色を明示。
- スマホの相談見出しを縮小し、改行を整える。
- QAへタブの画面外配置、パネル切れ、ボタン文字色の検査を追加。

## 第4段階: 公開前の最終判定

第3段階修正後の実画面を、発注者、クリエイティブディレクター、UX担当、実装担当、厳しい発注者の5視点で最終確認する。
以下がすべて成立した場合のみmainへ統合する。

- 最初の5秒で制作力と主サービスが伝わる。
- 立体表現が正常に動き、非対応環境では代替表示される。
- 全文と全デモがPC・タブレット・スマホで切れずに見える。
- アニメ調プロフィール画像が一枚だけ表示される。
- Instagram相談、相談テンプレートコピー、固定スマホ導線が機能する。
- 横スクロール、404、JavaScriptエラー、外部スクリプトがない。
- QA成功後のスクリーンショットを人間が確認する。
