# PAOPAO WEB PRODUCTION SKILL SYSTEM

この文書は、PAOPAOポートフォリオを場当たり的に修正せず、工程ごとの専門スキルを組み合わせて改善するための制作体系です。

## 全体フロー

`事実整理 → 読者設計 → 情報設計 → コピー → UI → モーション・メディア → 実装 → CMS・安全性 → QA → 人間レビュー → 公開`

各工程は独立して確認するが、最終成果物は一体として評価する。

---

## Skill 01: ファクト・エビデンス管理

### 目的
掲載可能な事実と、未検証・制作中の内容を分離する。

### 入力
- 職務経歴
- ユーザーが明示したプロフィール
- 公開してよい数値
- 現在の活動状況

### 出力
- 掲載可能事実一覧
- 使用禁止・要確認項目
- 実績と現在進行中の活動の区分

### 合格基準
- 数字の出所を説明できる
- 架空の顧客、案件、評価がない
- 学習中の内容を実績扱いしていない

---

## Skill 02: 読者・依頼者設計

### 目的
誰が何を判断するページかを明確にする。

### 主な読者
- 経営者
- 創業者
- 現場マネージャー
- 採用担当者
- フリーランサー
- 独立事業者

### 出力
- 読者別の知りたい情報
- 離脱しやすい箇所
- 相談判断に必要な材料

### 合格基準
- 10秒で「誰で、何を任せられるか」が分かる
- 読者が自分の課題と接続できる
- サイト制作者の自己満足ではなく、依頼者の判断材料になっている

---

## Skill 03: 情報設計・ナビゲーション

### 目的
情報量を削るだけでなく、必要な情報へ迷わず到達できる構造をつくる。

### 出力
- HOME / WORK / EXPERIENCE / PROCESS / CURRENT / JOURNAL / CONTACTの役割分担
- ページごとの優先情報
- 重複削減
- URLで直接共有できる構造

### 合格基準
- 一つのページへ全情報を積んでいない
- 同じ説明が複数ページで繰り返されていない
- 現在地と戻り先が分かる
- PC・スマホのナビゲーションが成立する

---

## Skill 04: 日本語コピー・校正

### 目的
日本語として正しいだけでなく、現代のビジネス話者が自然に読める文章へ整える。

### チェック軸
- 主語と対象が明確か
- 読者へ不自然に命令していないか
- 抽象語が担当内容を隠していないか
- 教科書、自己啓発、研修資料の口調になっていないか
- 英語見出しが日本語の理解を妨げていないか
- 同じ語尾・助詞・表現が続いていないか

### 合格基準
- 声に出して読んでも不自然でない
- 見出しだけで要点が伝わる
- 文章を読んだ後に「結局何？」が残らない
- 誇張せず、弱々しくもない

---

## Skill 05: コンバージョン設計

### 目的
閲覧者が、理解から相談まで迷わず進めるようにする。

### 出力
- 各ページから次に読むページへの導線
- Instagram DMへのCTA
- 相談時に伝える内容
- 連絡手段の視覚的識別

### 合格基準
- CTAが多すぎない
- Instagramロゴと文言が一致する
- 相談前に必要な情報が揃っている
- 連絡先を探し回らなくてよい

---

## Skill 06: UI・アートディレクション

### 目的
PAOPAOらしい世界観と、ビジネス上の信頼感を両立する。

### 方向性
- 墨色、朱、青緑、生成り
- 現代的に抽象化した日本要素
- 編集的な大見出しと明確なグリッド
- 情報の重要度に応じた密度差

### 合格基準
- 装飾が内容より先に目立たない
- 余白、文字サイズ、コントラストに一貫性がある
- カードを並べただけのテンプレート感がない
- 写真が背景から浮いていない
- PCとスマホを単純な縮小関係にしていない

---

## Skill 07: フロー・データ表現

### 目的
順序、因果、判断、推移を視覚的に理解させる。

### 使用対象
- 仕事の進め方
- 経歴の時系列
- 制作プロセス
- 実績値の比較

### 合格基準
- 順序のない情報を無理にフロー化していない
- 矢印や線の意味が分かる
- 注釈が本文の繰り返しではない
- スマホで読む順番が崩れない

---

## Skill 08: モーション・動画・背景画像

### 目的
世界観、奥行き、ページ遷移の理解を補助する。

### 原則
- オープニングは斬撃からページが開く連続演出
- ページ切り替えは短く、操作を妨げない
- 背景動画は無音、自動再生、ループ、playsinline
- スマホ・低負荷設定では静止画へ切り替える
- 文字の背後には可読性確保のレイヤーを置く

### 合格基準
- 動画が再生できなくてもページが成立する
- motion-reduceに対応する
- 読み込み中に大きなレイアウト移動がない
- 動画が本文や本人写真と競合しない
- 見栄えだけのために巨大データを埋め込まない

---

## Skill 09: フロントエンド実装

### 目的
設計と見た目を、壊れにくく保守可能なコードへ落とす。

### 対象
- HTMLの意味構造
- CSS設計
- ページルーティング
- アニメーション
- 画像・動画のフォールバック
- エラー処理

### 合格基準
- JavaScriptが無効・失敗しても最低限の情報へ到達できる
- 同一IDや重複初期化がない
- 固定コミットURLの依存関係が破綻していない
- コンソールエラーがない
- 横スクロールが発生しない

---

## Skill 10: レスポンシブ・アクセシビリティ

### 目的
端末、入力方法、身体条件によらず内容へ到達できるようにする。

### 合格基準
- 390px、768px前後、1024px、1440pxで確認
- タップ領域が狭すぎない
- 横スクロール型ナビが操作できる
- キーボードで移動できる
- フォーカス表示がある
- 色だけで状態を示さない
- alt、aria、見出し階層が適切
- 文字コントラストが不足していない

---

## Skill 11: JOURNAL・CMS・セキュリティ

### 目的
記事を安全かつ簡単に作成・更新・非公開化する。

### 現行方式
- GitHub Issuesを記事データとして使用
- PATによる管理者操作
- 新規作成、編集、Issueを閉じる形での公開停止

### 合格基準
- PATをソース、チャット、Issue、コミットへ保存しない
- 公開画面に管理UIを出さない
- 401 / 403 / 404 / 422を具体的に案内する
- 旧記事形式も読み込める
- ラベル有無だけに依存しない
- 保存後に公開一覧へ反映される
- 失敗時に入力内容が消えない

### 将来設計
OAuthやバックエンドCMSへの移行は、静的サイトの小修正として扱わず、認証・データ・権限・運用を含む別プロジェクトとして設計する。

---

## Skill 12: QA・リリース管理

### 目的
内部では動いている、ではなく、利用者の端末で成立する状態を確認する。

### 必須検証
- 公開URLを新規タブで開く
- オープニング終了
- ページ切り替え
- 戻る・進む
- 写真表示
- 動画と静止画フォールバック
- JOURNAL取得
- 新規保存、編集、公開停止
- Instagramリンク
- PC・スマホの横幅
- コンソールとネットワークエラー

### 合格基準
- 未検証事項を明記する
- スクリーンショットと実際の公開版が一致する
- ユーザーの実機スクリーンショットを最優先の証拠として扱う
- 固定URLに最新コミットSHAを使用する

## 人間レビューの実行方法

各人格は、AIの採点項目ではなく「実際にその立場でページを見た人」として反応を出す。

最低限、各人について次を記録する。

- 最初に目に入ったもの
- 10秒後に理解した内容
- 分からなかった言葉
- 信頼できた根拠
- 疑った表現
- 相談・採用・協業へ進むか
- 離脱するならどこか

レビュー結果は内部修正に使い、公開ページに人格名、点数、レビュー欄を出さない。

## スキル運用ルール

1. 変更依頼を受けたら、関係するSkillを特定する
2. 変更前に事実・目的・対象ページを確認する
3. 一つの問題を直すために、別の完成部分を壊さない
4. 実装後に関係Skillの合格基準で確認する
5. `docs/RELEASE_CHECKLIST.md`を通す
6. 変更理由と未検証事項をコミットまたはプロジェクト記録へ残す
