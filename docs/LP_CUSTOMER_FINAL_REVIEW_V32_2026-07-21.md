# PAOPAO LP V32 最終公開レビュー

対象: PR #14 / `agent/customer-first-lp-v32`

## 公開判定の前提

このLPの目的は、技術力を自慢することではない。営業DMや紹介でURLを受け取った個人・店舗・小規模事業者が、次の判断を短時間で行える状態を作ることである。

1. 自分のような相手を対象にしている。
2. 内容がまとまっていなくても相談できる。
3. 文章、構成、デザイン、公開まで頼める。
4. 相談前の不安と条件を確認できる。
5. この制作者へ一度相談する理由がある。

## 第4段階: 5視点の最終確認

### 1. 営業DMをスマホで開く店舗オーナー

確認結果:

- 最初の画面で「個人・店舗・小規模事業者向け」と明示されている。
- 主見出しは「言葉にできない魅力を、相談につながるホームページに。」であり、制作技術ではなく顧客が得る結果から始まる。
- 内容が未整理でも相談できること、文章整理から公開まで対応すること、無料モニター枠があることを初期画面で確認できる。
- スマホ下部に「変わり方・制作内容・相談する」の固定導線がある。

判定: 営業URLとして使用可能。

### 2. 初めてWeb制作を外注する個人事業主

確認結果:

- よくある悩み、改善前後、受け取れるもの、4段階の進め方、無料モニター条件、FAQ、相談時に送る3項目が順番に説明されている。
- 無料モニターについて、実績公開、対象規模、有料サービス等の実費、内容による有料提案・受付不可の可能性を明記している。
- 架空の改善例は「架空例」と明記し、実績の捏造に見えない。
- InstagramのDMへ送る相談テンプレートをコピーできる。

判定: 条件と相談方法を理解できる。

### 3. 複数の制作者を比較する小規模企業担当者

確認結果:

- 主サービスをホームページ・LP制作に固定し、自動化は問い合わせ後の作業を軽くする追加対応として整理している。
- 納品内容を「言葉・構成・表現・公開後に使える状態」の4点で具体化している。
- 経歴は年数の羅列ではなく、質問の先回り、顧客体験、分かりやすい説明という制作上の価値へ変換している。
- 「このサイト自体」を、悩み・設計・表現・検証の順で作った制作事例として説明している。

判定: 他者との比較に必要な違いと対応範囲が分かる。

### 4. クリエイティブディレクター

確認結果:

- 立体表現は抽象物ではなく、ホームページの「伝える・信頼・行動・運用」を示す4層のサイト設計図である。
- 立体図、改善前後デモ、納品工程図、Case 00の設計図が、すべて「情報を整理して相談へつなげる」という同じ意図で統一されている。
- アイボリー、濃紺、青、朱色、明朝体とゴシック体の組み合わせに一貫性がある。
- PCとスマホの日本語見出しは、意図しない一文字落ちを修正済み。
- 最終コピーと追加CSSはHTML/CSSソースへ直接移し、JavaScriptによる文章差し替えを廃止した。

判定: 表現にサービス上の意味があり、装飾の自己目的化を避けている。

### 5. フロントエンド・UX監査

最終ソース移行前の同一表示に対する確認結果:

- PC 1440px、タブレット1024px、スマホ390pxでVisual QAとPerformance QAが成功。
- ready time: PC 346ms / タブレット312ms / スマホ244ms。
- frame pacing p95: PC 22.3ms / タブレット16.8ms / スマホ16.7ms。
- 改善前後の切り替え: PC 7.5ms / タブレット5.4ms / スマホ6.3ms。
- 外部JavaScript、Canvas、横スクロール、同一オリジン404、コンソールエラー、ページエラー、ローカル通信失敗は0件。
- 4層立体図、改善前後、FAQ、相談テンプレート、固定ヘッダー後のアンカー位置を自動検査している。
- 最終ソース移行後に同じVisual QAとPerformance QAが成功することを、main統合の必須条件とする。

## 最終公開ゲート

次の条件をすべて満たした場合のみmainへ統合する。

- 最終ソースに対するVisual QA成功。
- 最終ソースに対するPerformance QA成功。
- PC、タブレット、スマホのスクリーンショットで、見出し欠け・一文字落ち・空白化・パネル切れがない。
- `index.html`に最終コピーと追加CSSが直接記述されている。
- `v32-customer-first.js`に文章差し替え処理やCSS動的読み込みが残っていない。
- 公開後のGitHub Pagesデプロイが成功し、デプロイ元SHAがmainのマージSHAと一致する。

この文書の追加後にGitHub Actionsを再実行し、成功した場合に限り公開可と判定する。
